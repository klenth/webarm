WebARM

    An ARM32 assembler/simulator for education

    Using WebARM, you can easily write and execute 32-bit ARM code from within your browser. This page documents the ARM32 assembly dialect used in WebARM.

    Structure

    In a WebARM program, each line consists of the following:

    1. (optional) a symbol, starting with a period, that can be used to refer to this location in the program;
    2. either an instruction (one operation for the processor to perform) or a directive (instructing the assembler to modify the generated code in some fashion).

    Some examples: 

            ; A symbol on a line by itself (will refer to the *next* nonempty line)
.top
        ; A single instruction
        ADD     R0, R0, R2
        ; (So the symbol ".top" refers to the ADD instruction)

        ; Another instruction
        B       .main

        ; The next line defines a symbol (.msg) and uses the DCB directive to include
        ; some data (in this case, text) directly within the program. The ", 0" at the
        ; end includes a zero byte (NUL terminator) to mark the end of the string.
.msg    DCB     'Hello, world!', 0

        ; A directive: ALIGN inserts any zero padding necessary to ensure that the next
        ; stuff will start at a multiple of 4 bytes within the file (necessary for
        ; instructions).
        ALIGN

        ; A symbol (.main) referring to a LDR instruction
.main   LDR     R1, =123_456

    Comments

    Comments begin with a ; and extend to the end of the line. There is no equivalent of the C-style /* ... */ block comment.

    Spacing

    Unlike most high-level languages, code is not allowed to cross lines (you cannot split an instruction or directive across lines).

    Aside from this, spacing is arbitrary. It is not required to follow spacing similar to what is shown above (symbols at the far left, instructions indented once, operands indented one more time), but it does make the code easier to read.

    Case sensitivity

    Instructions, directives, and the names of registers are case-insensitive (mov r0, pc is the same as MOV R0, PC), but symbol names are case-sensitive (.msg and .MSG are different symbols).

    Registers

    ARM32 has 16 registers: R0 through R15. The last several registers serve special purposes and have alternate names:

    Register Alternate Description
    R13 SP Stack pointer (holds the address of the top of stack)
    R14 LR Link register (the return address of the current function)
    R15 PC Program counter (the address of the instruction to be executed next)

    Numbers

    In all places where numbers are expected — for example, in immediate values (such as #123) — they can be expressed in decimal (123), hexadecimal (0x7b), or binary (0b1111011). You can use _ to group digits within numbers (like 0b111_1011) with no impact on the value.

    Strings

    The DCB directive allows you to enter strings whose characters (bytes) are directly included in the program; this is the only place in WebARM where strings can be used. These strings can be delimited by either single quotes ('...') or double quotes ("..."), and in either case the usual escape sequences \n, \', \", \\, \t, and \0 are recognized.

    Condition codes

    Nearly all ARM instructions can take a condition code suffix: a two-letter abbreviation specifying that the instruction should only be performed if some condition is met. These conditions are tested on the basis of the NZCV (Negative, Zero, Carry, oVerflow) bits, which can be set by various instructions. The condition codes are as follows:

    Code Meaning Condition
    EQ Equal (or zero) Z=1
    NE Not equal (or nonzero) Z=0
    CS Carry set (unsigned ≥; unsigned overflow) C=1
    CC Carry clear (unsigned <; no unsigned overflow) C=0
    MI Negative N=1
    PL Positive or zero N=0
    VS oVerflow set (signed overflow) V=1
    VC oVerflow clear (no signed overflow) V=0
    HI Unsigned > C=1 and Z=0
    LS Unsigned ≤ C=0 and Z=1
    GE Signed ≥ N=V
    LT Signed < N≠V
    GT Signed > Z=0 and N=V
    LE Signed ≤ Z=1 or N≠V
    AL Always (none)

    When no code is specified, it is the same as using AL (always).

    The S suffix

    Many instructions (specifically data processing and multiplication) have an optional S suffix; when S is included, then the instruction sets the NZCV bits according to the result. For most instructions, this means the following:

    • N is set if the result is negative when interpreted as a signed integer, i.e., if the top bit is 1.
    • Z is set if the result is zero, that is, if all bits of the result are zero.
    • C is set if there was a carry out from the last digit; for addition and subtraction, this happens when there was an overflow when the numbers are interpreted as unsigned.
    • V is set if there was overflow when the numbers are interpreted as signed.

    An instruction can have both a condition code and the S suffix; in that case, they can be in either order (ADDSEQ and ADDEQS mean the same). Some instructions (CMP, CMN, TST, TEQ) exist only to set NZCV, so they do not require the S suffix.

    Examples

            ; Subtract the values in R4 and R5 and set NZCV according to the result
        SUBS    R0, R4, R5

        ; Add, but only if Z=0 (that is, if R4 != R5)
        ADDNE   R0, R1, R1

        ; Add, but only when N=0; set the bits of NZCV according to the result
        ADDSPL  R2, R1, R1
        ; This is the same:
        ADDPLS  R2, R1, R1
    Data processing instructions

    These are the instructions used to do most calculations.

    The S form

    Most of these instructions can be suffixed with S — for example, ADD becomes ADDS — which instructs them to set the NZCV bits according to the result. (See condition codes for more information.) This is notated in the table below by [S] after instructions that accept it.

    The only instructions in this section that don’t have an S form are the ones that exist specifically to set NZCV, and thus writing S would be redundant: CMP, CMN, TST, and TEQ.

    Condition codes

    Like nearly all ARM instructions, all of these instructions can be suffixed with a condition code to execute them conditionally. Not including a condition code is the same as specifying AL (“always”). These can be combined with S in either order.

    Forms of operand2

    All of these instructions have an operand2 that can take various forms:

    • imm: an immediate value, for example #123

      Note that most numbers will not fit in the immediate field of one of these instructions: specifically, only those that can be obtained by rotating a sign-extended 8-bit value by an even number of bits within a 32-bit field. This includes powers of two and many common numbers, but most numbers do not work. For these numbers, you will need to load or compute it into a register first and then use that register as an operand.

    • sym: a symbol (like .loop_top), or an expression involving symbols (like 2*(.loop_top - .))

      Symbolic operands are subject to the same limitations as immediates mentioned above.

    • reg: a register
    • reg, shift imm: the value of a register, shifted by some immediate amount. shift can be any of LSR, ASR, LSL, ASL (a synonym for LSL), or ROR. For example, R4, LSL #1 means to take the value in R4 and shift it left by one bit.
    • reg, shift reg: the value of a register, shifted by an amount in another register; shift has the same options as above. For example, R4, LSL R5 means to take the value in R4 and shift it left by a number of bits in R5.

    Instructions

    In the table below, Rd is the destination register (where the result will go), Rm is an operand register (one of the inputs to the operation), and operand2 is as described above.

    Instruction syntax Operation Notes
    AND[S] Rd, Rm, operand2 Rd ← Rm AND operand2 Same as the & operation in C
    EOR[S] Rd, Rm, operand2 Rd ← Rm XOR operand2 Same as the ^ operation in C
    SUB[S] Rd, Rm, operand2 Rd ← Rm - operand2
    RSB[S] Rd, Rm, operand2 Rd ← operand2 - Rm “RSB” is short for “Reverse SuBtract”
    ADD[S] Rd, Rm, operand2 Rd ← Rm + operand2
    ADC[S] Rd, Rm, operand2 Rd ← Rm + operand2 + C Includes the value of the C bit (from NZCV) in the addition
    SBC[S] Rd, Rm, operand2 Rd ← Rm - operand2 + C - 1
    RSC[S] Rd, Rm, operand2 Rd ← operand2 - Rm + C - 1
    TST Rm, operand2 (nothing) ← Rm AND operand2 Sets the bits of NZCV
    TEQ Rm, operand2 (nothing) ← Rm XOR operand2 Sets the bits of NZCV
    CMP Rm, operand2 (nothing) ← Rm - operand2 Sets the bits of NZCV
    CMN Rm, operand2 (nothing) ← Rm + operand2 Sets the bits of NZCV
    ORR[S] Rd, Rm, operand2 Rd ← Rm OR operand2 Same as the | operator in C
    MOV[S] Rd, operand2 Rd ← operand2 Copies a value into a register
    BIC[S] Rd, Rm, operand2 Rd ← Rm AND NOT operand2 “BIC” is short for “BIt Clear”
    MVN[S] Rd, operand2 Rd ← NOT operand2 Same as the ~ operator in C

    Examples

            ; Add the values in registers R4 and R12, putting the result back in R4 (R4 += R12)
        ADD     R4, R4, R12

        ; Decrease the value of R10 by 1, setting the bits of NZCV
        SUBS    R10, R10, #1

        ; If the result was negative, set R10 back to zero
        MOVLT   R10, #0

        ; Test whether the value in R0 has bit 8 set
        TST     R0, #0x100

        ; If it was, then OR it with the value in R5, shifted right by 8 bits; set the bits
        ; of NZCV according to the result
        ORRSNE  R0, R0, R5, ASR #8

    Shift “instructions”: LSL, ASL, LSR, ASR, ROR

    ARM does not have separate instructions for shifts; these are merely aliases for MOV instructions with an appropriately shifted operand2 (for example, LSL R0, R1, #2 translates to MOV R0, R1, LSL #2). Consequently they cannot take a shifted operand, since the shifter is already in use. However, S and condition codes can still be used.

    Instruction syntax Operation Notes
    LSL[S] Rd, Rm, imm
    ASL[S] Rd, Rm, imm
    LSL[S] Rd, Rm, Rn
    ASL[S] Rd, Rm, Rn     		Rd ← Rm LSL imm
    Rd ← Rm LSL Rn     		Logical left shift (same as << in C). (ASL is a synonym for LSL).
    LSR[S] Rd, Rm, imm
    LSR[S] Rd, Rm, Rn     		Rd ← Rm LSR imm
    Rd ← Rm LSR Rn     		Logical right shift (same as >> of an unsigned int in C).
    ASR[S] Rd, Rm, imm
    ASR[S] Rd, Rm, Rn     		Rd ← Rm ASR imm
    Rd ← Rm ASR Rn     		Arithmetic right shift (same as >> of a signed int in C).
    ROR[S] Rd, Rm, imm
    ROR[S] Rd, Rm, Rn     		Rd ← Rm ROR imm
    Rd ← Rm ROR Rn     		Right rotate (like LSR, but the bits shifted off the right end appear on the left)
    Multiplication instructions

    ARM32 has two multiplication instructions that perform multiplication. In general the product of two 32-bit numbers would be a 64-bit number; these instructions keep only the bottom 32 bits of the result, so that they function similarly to the integer multiplication operator in C.

    The S form

    These instructions can be suffixed with S — for example, MUL becomes MULS — which instructs them to set the NZCV bits according to the result. (See condition codes for more information.) This is notated in the table below by [S] after instructions that accept it.

    When used with S, multiply instructions set the N and Z bits but leave C and V unchanged.

    Condition codes

    Like nearly all ARM instructions, all of these instructions can be suffixed with a condition code to execute them conditionally. Not including a condition code is the same as specifying AL (“always”). These can be combined with S in either order.

    Operands

    Unlike the data processing instructions, multiplication instructions do not support flexible operands (immediates, symbols, and shifted registers). Operands to multiplication instructions can only be registers.

    Instructions

    In the table below, Rd is the destination register (where the result will go), and Rm, Rn, and Ro are operand registers (the inputs to the operation).

    Instruction syntax Operation Notes
    MUL[S] Rd, Rm, Rn Rd ← Rm × Rn Same as integer multiplication in C
    MLA[S] Rd, Rm, Rn, Ro Rd ← Rm × Rn + Ro “MLA” is short for “multiply and accumulate”

    Examples

            ; Calculate R4² and put the result in R5
        MUL     R5, R4, R4

        ; Calculate R2×R3 + R4 and put the result in R1, setting N and Z according to
        ; the result, but only if the last comparison had the result “less than”
        MLALTS  R1, R2, R3, R4
    Branching instructions

    Branch instructions transfer control to another location in the program; in other words, they change the value of R15 (also known as PC). In most cases the argument to a branch instruction will be a symbol (meaning “go execute the instruction there next”) rather than an immediate.

    Condition codes

    Like nearly all ARM instructions, all of these instructions can be suffixed with a condition code to execute them conditionally. Not including a condition code is the same as specifying AL (“always”).

    Branch with link

    The B and BX instructions merely branch; the BL instruction additionally saves the current PC in the register LR (Link Register, a.k.a. R14) before branching. This saved address can be used to return to the point that was branched from, so normally BL is used for function calls while B is used for flow control within a function (if statements, loops, etc.).

    B is frequently combined with a condition code (BEQ, BLT, etc.).

    Instructions

    In the table below, imm is an immediate value (a number, such as #123), sym is a symbol or symbolic expression (such as .loop_top or 4 * (.loop_top - .)), and reg is a register.

    Instruction syntax Operation Notes
    B imm Branch to address imm bytes from PC Relative branch: jumps forward or backward from current instruction
    B sym Branch to address sym Absolute branch: jumps to the address of the symbol
    BL imm Save current value of PC in LR, then branch to address imm bytes from PC Relative branch with link: jumps forward or backward from current instruction and allows for return
    BL sym Save current value of PC in LR, then branch to address sym Absolute branch with link: jumps to the address of the symbol and allows for return
    BX reg Branch to the address in register reg Absolute branch (there is no “with link” version available)

    Examples

            ; Branch to a symbol (unconditionally)
        B       .loop_top

        ; Skip the next instruction (branch to PC+4)
        B       #4

        ; Compare the values in registers R0 and R1 and branch if R0 < R1
        CMP     R0, R1
        BLT     .end

        ; Call a function (using BL to save the return address in LR)
        BL      .shuffle
    Memory instructions

    Memory access is vital to any nontrivial program. Loads (instructions that read values from memory) and stores (instructions that write values to memory) are complex and consequently the syntax is also complex.

    Single data transfer (LDR, STR)

    LDR and STR load and store (respectively) a single value, either a word (4 bytes / 32 bits) or a byte, into or from a register. In the table below, placeholders have the following meanings:

    • num: a number (such as 123 or 0xff);
    • imm: an immediate (such as #123);
    • sym: a symbol or symbolic expression (like .array or .array + 4);
    • reg: a register;
    • ±reg: a register with an optional + or – in front of it;
    • shift: a shift operation (LSL, ASL, LSR, LSR, or ROR).
    Instruction syntax Operation/notes
    LDR reg, =num Loads the given 32-bit number into reg. (Differs from MOV reg, imm in that num can be any value that fits in 32 bits.)
    LDR  reg1, [reg2]
    LDRB reg1, [reg2]     		Loads one word (LDR) or one byte (LDRB) from memory address reg2 into register reg1. (For LDRB, the bottom byte of reg1 gets the loaded value; the other three bytes are set to zero.)
    STR  reg1, [reg2]
    STRB reg1, [reg2]     		Stores one word (STR) or one byte (STRB) from register reg1 into memory at address reg2. (For STRB, the bottom byte of reg1 is stored; the other three bytes are ignored.)
    LDR  reg1, [reg2, imm]
    LDRB reg1, [reg2, imm]
    STR  reg1, [reg2, imm]
    STRB reg1, [reg2, imm]     		Loads/stores from/to memory address reg2+imm into/from register reg1. This (and other LDR/STR variants with [reg, something]) are referred to as preindexed addressing.
    If a ! is placed at the end, also updates reg2 ← reg2 + imm. This is useful when iterating over arrays.
    LDR  reg1, [reg2, ±reg3]
    LDRB reg1, [reg2, ±reg3]
    STR  reg1, [reg2, ±reg3]
    STRB reg1, [reg2, ±reg3]     		Loads/stores from/to memory address reg2 ± reg3 into/from register reg1.
    If a ! is placed at the end, also updates reg2 ← reg2 ± reg3. This is useful when iterating over arrays.
    LDR  reg1, [reg2, ±reg3, shift imm]
    LDRB reg1, [reg2, ±reg3, shift imm]
    STR  reg1, [reg2, ±reg3, shift imm]
    STRB reg1, [reg2, ±reg3, shift imm]     		Loads/stores from/to memory address reg2 ± (reg3 shift imm) into/from register reg1.
    If a ! is placed at the end, also updates reg2 ← reg2 ± (reg3 shift imm). This is useful when iterating over arrays.
    LDR  reg1, [reg2], imm
    LDRB reg1, [reg2], imm
    STR  reg1, [reg2], imm
    STRB reg1, [reg2], imm     		Loads/stores from/to memory address reg2 into/from register reg1, then updates reg2 ← reg2 + imm. This (and other LDR/STR variants with [reg], something) are referred to as postindexed addressing.
    LDR  reg1, [reg2], ±reg3
    LDRB reg1, [reg2], ±reg3
    STR  reg1, [reg2], ±reg3
    STRB reg1, [reg2], ±reg3     		Loads/stores from/to memory address reg2 into/from register reg1, then updates reg2 ← reg2 ± reg3.
    LDR  reg1, [reg2], ±reg3, shift imm
    LDRB reg1, [reg2], ±reg3, shift imm
    STR  reg1, [reg2], ±reg3, shift imm
    STRB reg1, [reg2], ±reg3, shift imm     		Loads/stores from/to memory address reg2 into/from register reg1, then updates reg2 ← reg2 ± (reg3 shift imm).

    Preindexed and postindexed addressing

    Most of the complexity of LDR/STR instructions comes from preindexed (e.g. LDR R1, [R2, #4], possibly followed by !) versus postindexed (LDR R1, [R2], #4) addressing. It is never necessary to use these — you can always accomplish the same task without pre/postindexing by adding or subtracting before or after. Using pre/postindexing allows you to access an array element and also move to the next (or previous) index in the same instruction.

    Pre/postindex instruction Equivalent without pre/postindexing Analog in C
    LDR R1, [R0, #4] ADD R12, R0, #4
    LDR R1, [R12]    †    		 array[i + 1]
    LDR R1, [R0, #4]! ADD R0, R0, #4
    LDR R1, [R0]    		 array[++i]
    LDR R1, [R0], #4 LDR R1, [R0]
    ADD R0, R0, #4    		 array[i++]

    †R12 here just means any register that is otherwise unused (to act as a temporary for R0+4).

    Examples

            ; Load the address .data into R0
        LDR     R0, =.data

        ; Load the value at that address (data[0]) into R5
        LDR     R5, [R0]

        ; Load the next value (data[1]) into R6
        LDR     R6, [R0, #4]

        ; Load the first value (data[0]) into R7, and after that move R0 to index 3
        LDR     R7, [R0], #12

        ; Save the value in R2 into the next element (data[4]), also moving R0 to point at it
        STR     R2, [R0, #4]!

        ; Set R0 back to the beginning of the array
        LDR     R0, =.data

        ; Load the number 150 into R1
        LDR     R1, =150

        ; Load the value of data[150] into R8
        LDR     R8, [R0, R1, LSL #2]

        ; Set R0 to index 100 of the array
        LDR     R0, =.data + 4*100

        ; Store the value in R11 to index 100-R4 in the array; update R0 to point at it
        STR     R11, [R0, -R4, LSL #2]!

    Block data transfer (LDM, STM)

    ARM also provides instructions for loading/storing to/from multiple registers simultaneously, allowing the programmer to transfer large amounts of data between registers and memory all at once. This can be used to copy data quickly from one place to another (e.g. copy 12 words at a time from memory into R0...R11, then copy those registers’ values back into memory somewhere else) but is more frequently used for pushing onto and popping off of the stack (whose address is usually kept in SP, also known as R13).

    The operands to both LDM and STM are

    • baseReg: the base register for the load/store, holding the address in memory where values will be loaded/stored. Putting ! after the register writes the final address back into baseReg at the end.
    • dir: a two-letter code specifying what direction in memory the registers will be stored in (see below).
    • registers: any combination of the 16 registers. They can be specified individually ({R1,R2,R3,R4}), with a range ({R1-R4}), or mixed ({R1-R4,R6-R9,R12}). These are the registers that will be loaded into or stored from.
    Instruction syntax Operation
    LDMdir baseReg, registers Load values for registers from memory starting at address baseReg
    STMdir baseReg, registers Store the values in registers to memory starting at address baseReg
    The directions are as follows. There are only four options (preincrement, postincrement, predecrement, postdecrement), but each has two names, depending on whether they are being interpreted as stack operations or non-stack and whether it is a load or a store.
    Stack Non-stack Operation Resulting addresses
    LDMED, STMFA LDMIB, STMIB Preincrement load/store baseReg+4, baseReg+8, baseReg+12, ...
    LDMFD, STMEA LDMIA, STMIA Postincrement load/store baseReg, baseReg+4, baseReg+8, ...
    LDMEA, STMFD LDMDB, STMDB Predecrement load/store baseReg-4, baseReg-8, baseReg-12, ...
    LDMFA, STMED LDMDA, STMDA Postdecrement load/store baseReg, baseReg-4, baseReg-8, ...

    Directions

    To interpret these directions, first look at the “Non-stack” column. The two letters at the end specify whether we are incrementing the memory address between each register (I) or decrementing it (D) — that is, whether memory addresses are increasing (I) or decreasing (D) as each register is loaded/stored — and whether that increment/decrement is applied before (B) or after (A) each register is loaded/stored. For example, LDMIB means to increment before each register, so the first register’s value will be loaded from baseReg+4, the second from baseReg+8, and so on; LDMIA means the address is incremented after each register, so the first register is loaded from address baseReg, the second from baseReg+4, etc. The non-stack directions are the same whether you are doing a load or a store.

    In the stack-based directions, the letter E/F is whether the stack is initially “empty” or “full” — that is, whether the address in baseReg does not hold a value on the stack (E ) or does (F); this determines whether in a store we should postincrement/decrement (E, since there is not already a value there, we should store and then increment/decrement) or preincrement/decrement (F, since there is already a value at address baseReg, we should increment/decrement first and then store). The A/D letter denotes whether the stack is “ascending” (grows towards higher memory addresses) or “descending” (grows towards lower memory addresses). The stack-based directions are reversed between loads and stores: LDMED is the same as LDMIB, but STMED is the same as STMDA, the opposite direction; this is because undoing a preincrement load requires a postdecrement store, and so on.

    Regardless of direction, the lowest register maps to the least (not last!) memory address. R0 will always correspond to an address less than R1 if both are included in the register set.

    Examples

            ; Copy the eight values at addresses R10, R10+4, ..., R10+28 into R0, R1, ..., R7
        LDMIA   R10, {R0-R7}

        ; Store those same eight values back into memory, in the same order, starting at address R11;
        ; when done, set R11 to the final address (R10+32)
        STMIA   R11!, {R0-R7}

        ; Push the values of R5, R6, R7, and LR onto the stack. This is a common way to start an
        ; assembly function (this would allow the function to use R5, R6, R7, which are typically
        ; callee-save registers, and also preserve LR so that this function can itself call
        ; another function).
        STMFD   SP!, {R5-R7,LR}

        ; Pop R5, R6, R7 off the stack and return to the address that used to be in LR (instead of
        ; popping the fourth value back into LR, we pop it into PC, effectively branching back to
        ; the return address).
        LDMFD   SP!, {R5-R7,PC}
    Other instructions

    There are a number of other instructions that stand on their own, not being closely related to others.

    STOP

    Ends the program. This instruction can be suffixed with a condition code. (This is not a real ARM instruction, but merely a convenience provided by WebARM.)

    Instruction syntax Operation
    STOP End the program.

    BREAK

    Breaks the program, if running in debug mode; execution will be paused so that the programmer can inspect values of registers, memory, etc., and step forward or backward from that point. This instruction can be suffixed with a condition code. (This is not a real ARM instruction, but merely a convenience provided by WebARM.)

    Instruction syntax Operation
    BREAK Breaks the program, if running in debug mode.

    SWI

    Executes a software interrupt. On a real ARM system this invokes the operating system to execute a system call. On WebARM there is a small collection of syscall functions that can be executed in this fashion (documented elsewhere). This instruction can be suffixed with a condition code.

    Instruction syntax Operation
    SWI Executes a software interrupt, invoking a system call.

    NOP

    Does nothing (“no-op”). This instruction can not be suffixed with a condition code.

    Instruction syntax Operation
    NOP Does nothing.
    Directives

    Directives instruct the assembler to do something, but do not directly generate executable code. They are mostly useful for including data other than instructions in the executable, for example an array of data or some text.

    DCD: Define Constant Data

    DCD inserts some words (4-byte/32-bit values) directly into the assembled code. It automatically aligns to a 4-byte boundary (that is, if the current address isn’t a multiple of 4, it will insert zeros for padding). DCD is mostly used when there is a collection of data to be included in the program (such as initial values for an array).

    The values in a DCD can be in decimal, hexadecimal (0x...), or binary (0b...). They can also be symbols or symbolic expressions.

    Examples

            ; Insert some data for a program
        ; (Similar to "int data[] = { 1, 2, 3, 0xff, 100 };")
.data   DCD     1, 2, 3, 0xff, 100

        ; The first word is the address of .data above; the second is
        ; the address of *this* data, .data2; the third is the difference
        ; between the two, divided by 4
.data2  DCD     .data, ., (. - .data)/4

    DCB: Define Constant Bytes

    DCB is similar to DCD except that the values given are bytes (either signed or unsigned: in the range -127 to +255) rather than words. Alternatively, you can give a string, in which case each character of the string becomes one byte.

    Examples

            ; A welcome message for a program (with a NUL terminator at the end)
.msg0   DCB     'Hello!',0

        ; Another way to include the same data (in ASCII, H is 0x48, etc.)
.msg1   DCB     0x48, 0x65, 0x6c, 0x6c, 0x6f, 0x21, 0x00

        ; Some bytes of data
.bytes  DCB     0x40, 0b1001_0110, -4, -0x20

    FILL

    A FILL directive inserts a specified number of zeros (or another byte value) into the code. This is helpful for allocating a large chunk of memory, for example, space for an array. The first argument is the number of bytes to fill (which can be a number or a symbolic expression), and the second argument is an optional byte value to fill.

    Examples

            ; 24 zeros (if not specified, the fill value is 0)
        FILL    24

        ; Room for 100 words
.array  FILL    4*100

        ; 1000 Fs, for a day when you're feeling frustrated
.effs   FILL    500, 0xff

    ALIGN

    An ALIGN directive ensures that the next data (whether it be an instruction or another directive) begins at an address that is a multiple of some value (4 by default) by adding zero bytes if necessary. This is necessary in some cases as ARM cannot make unaligned memory accesses (for example, it cannot load bytes 3–6 as a single word since 3 is not a multiple of 4); since instructions are words, they must always start on a multiple of 4 as well, but directives like DCB and FILL can cause addresses to become unaligned. 

            ; A chunk of memory that will cause unalignment
        DCB     'Hello!',0   ; 7 bytes

        ; If we put an instruction here, it would not work since it would
        ; start at address 7, which is not a multiple of 4

        ALIGN
        ; Now it's OK to have an instruction, as ALIGN will bump us to address 8
        ADD     R1, R0, R0

        ; Align to a multiple of 256 (so the next address will be a multiple of
        ; 256 / 0x100)
        ALIGN   256
        DCD     1, 2, 3, 4
